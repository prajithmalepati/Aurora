# Visual Overhaul — Implementation Readiness Audit

This is an as-implemented gap analysis, not a design review. The UX critique (`2026-05-25-aurora-ux-critique.md`) already handled design. This document maps the design spec onto the actual codebase and identifies every concrete change needed before implementation can start.

## Blockers: Must Resolve Before Any Frontend Work

### B1. Backend song data model has no new columns

The spec requires three new columns on the `songs` table. They do not exist, and the code has no awareness of them.

**What's missing:**

| Pydantic model | `SongResponse` missing all three fields | `models.py:23-39` |
| Query helper | `song_row_to_dict()` doesn't SELECT or return any of them | `songs.py:21-56` |
| List songs query | `SELECT` in `list_songs()` missing columns | `songs.py:92-104` |

| Filter query | `SELECT` in `filter_songs()` missing columns | `filter_engine.py:121-133` |
| Filter result dicts | Manual dict construction missing fields | `filter_engine.py:148-162` |
| Playlist song queries (×4) | All 4 SELECT queries missing columns; all 4 `SongResponse(...)` constructions missing fields | `playlists.py:235-253,270-288` (get); `390-412,425-443` (delete song); `505-527,540-558` (reorder); `781-803,816-834` (add song) |

`song_row_to_dict()` is the central bottleneck — fixing it propagates the new fields to: `list_songs`, `get_song`, `update_song`, `assign_tags`, `remove_tag_from_song` (5 call sites). The playlists router (4 call sites) and filter engine (1 call site) construct song objects independently and need separate updates.

**Why it blocks:** Nothing in the spec works without these fields flowing through the API. The per-song color pipeline (Phase 3), the shader `uColor2` uniform (Phase 4), and the waveform bar (Phase 5) all depend on `dominant_color`, `dominant_color_2`, and `waveform_peaks` arriving in song objects.

### B3. Scanner can't produce what the API needs to serve

`extract_metadata()` in `file_scanner.py:142-185` returns `{title, artist, album, duration, file_path, file_format}`. No waveform peak extraction. No color extraction.

The spec proposes:
- **Waveform peaks:** `miniaudio` or `audioop` → 1000-bin normalize → JSON array
- **Dominant colors:** PIL 64×64 album art → K-means (scikit-learn) → filter near-gray/white/black → OKLCH clamp → store as string

Neither `miniaudio` nor `scikit-learn` is installed. `Pillow` is not in `backend/requirements.txt` (though it may be transitively available). The spec's "approx 1.5 hrs" estimate for this assumes these libraries work as expected on the first try. The actual work is:
1. Install and verify `miniaudio` can decode FLAC/Opus/M4A
2. Implement peak extraction with proper normalization (not all audio formats decode the same way)
3. Implement album art color extraction (PIL → numpy → K-means)
4. Integrate into `extract_metadata()` or a separate post-scan pipeline
5. Store in INSERT/REPLACE queries

### B5. `html5: true` → AnalyserNode bridge is unproven (highest technical risk)

This is the single most dangerous assumption in the spec. From GPT 5.5.1's review — and Kimi's analyser advice confirms the fragility.

`useAudioPlayer.ts` creates Howls with `html5: true` (line 83). This bypasses Web Audio entirely for decoding and routes audio through the browser's `<audio>` element pipeline. The spec says to tap `Howler.masterGain` to insert an `AnalyserNode`:

```ts
const ctx = Howler.ctx;
const analyser = ctx.createAnalyser();
Howler.masterGain.connect(analyser);
```

This assumes `Howler.masterGain` receives output when `html5: true`. According to Howler.js source, `masterGain` is connected via Web Audio when the codec is decoded in-API. But `html5: true` mode creates the Howl differently — the audio path may not flow through `masterGain` at all, leaving the analyser permanently silent.

**Risk:** The entire audio-reactive shader (transient-sensitive amplitude → `uAmplitude` → curtain pulsing) collapses silently. Weeks of GLSL work produce curtains that never respond to music.

**Fix required:** Before Phase 4 (shader) begins, prototype the analyser path with a minimal test: create a Howl with `html5: true`, connect an analyser, verify `getByteFrequencyData()` returns non-zero values. If it fails, the fallback options are: (a) use `html5: false` (Web Audio native decoding, higher latency for FLAC), (b) run a separate `AudioContext` + `MediaElementSource` from the Howler's internal `_sounds[0]._node`, or (c) drop audio-reactive curtains and keep `uAmplitude` at 0.

### B6. `songStore.view` can't drive Aurora intensity

The spec says `uIntensity` should be 15% in filter view, 20% in library, 80% in now-playing. The proposed source is `songStore.view`:

```typescript
// songStore.ts — view type
type View =
  | { kind: "all-songs" }
  | { kind: "filter" }
  | { kind: "playlist"; playlistId: number }
  | { kind: "settings" }
```

This encodes 4 states. The spec's calm regions contract requires knowledge of 5 states:
- Library / Song list → maps to `{kind: "all-songs"}`
- Filter / Query bar active → maps to `{kind: "filter"}`
- Playlist detail → maps to `{kind: "playlist"}`
- **Now-playing (expanded PlayerBar)** → **NO MAPPING EXISTS**
- **Idle (no song, no interaction 30s)** → **NO MAPPING EXISTS**

The PlayerBar expands when `currentSong !== null` (a `playerStore` value, not a `songStore.view` value). The idle state requires both "no current song" AND a 30-second inactivity timer. Neither lives in `songStore.view`.

**Fix required:** Either add the concept of focus/now-playing mode to the store, or derive `uIntensity` from three separate signals (view + currentSong + idle timer) in the shader hook rather than from `songStore.view` alone.

## Spec Internal Contradictions

These are places where the spec document contradicts itself or the actual code architecture.

**C1. Waveform pipeline described twice, two different ways.**

Section 2 "On song load" says: `fetch(songUrl) → arrayBuffer → AudioContext.decodeAudioData → sample peaks`. Section 3 "Per-Song Color Pipeline" says: "Backend scanner...: waveform_peaks on songs" and "Frontend (song load = zero computation, reads from API)." Phase 1 says "Pre-computed waveform peaks (server-side)." The spec keeps the frontend decode path as step 1 of the waveform section and the backend path as the color pipeline section. They need to be reconciled into a single pipeline description.

**C2. `uColor1` / `uColor2` routing is underspecified.**

Section 1 (shader spec): `uColor1` = fixed brand teal `oklch(0.72 0.18 185)`. `uColor2` = album dominant color fringe.

Section 3 (consumer list): "Shader `uColor2` ← `--song-color-2` (per-song fringe)." OK, consistent. But then: "Shader `uColor1` ← fixed brand teal — never overridden by album art." This is placed in the consumer list as if it comes from CSS, but there's no `--aurora-brand` or `--song-color-1` CSS variable defined for the fixed teal. It's just hardcoded in the shader as a `uColor1` uniform value. The consumer list should either define a CSS variable for `uColor1` or document that it's a shader constant, not a CSS variable.

**C3. `uIntensity` source (`songStore.view`) doesn't encode the data it needs.**

Documented above as B5. This is both a blocker and a spec error — it's in the implementation plan (Phase 4 step 9).

**C4. The play button spec references a version of the feature the spec itself killed.**

The original design spec (as recorded in HANDOFF.md Session 25) had "mouse waver + hold glow." The current spec (Section 4) explicitly rejects these: "Why not mouse waver or hold glow: hover rotation on a control used hundreds of times/day reads as 'demo,' not premium software. Hold glow adds mystery with no utility." The spec's current play button definition (state-driven star bloom) is the correct version. The question being asked references the older version that the spec already abandoned.

## Design Risks (Not Blockers, But Will Cause Rework)

These are judgments about what will feel wrong in practice. They are not blocking — you can ship without addressing them — but they represent design debt.

**R1. Global `uIntensity` without spatial masking.** The calm regions contract adds a global scalar per view. This works for the PlayBar and the content background, but the aurora canvas spans the full viewport including the sidebar, the search bar, and dense text columns. A spatial falloff (dimmer behind sidebar, brighter in empty screen margins) would prevent the "visual noise" problem the UX critique flagged. Without it, the 20% intensity in library view is still a shader animating behind 14px tabular text.

**R2. PlayerBar color overlap.** The PlayerBar will have (a) the aurora shader visible through the glass background, (b) `--song-color` at 5% alpha tint, and (c) the color bleed radial gradient at 35% opacity. On deeply saturated album colors (dark red, deep blue), the teal aurora + red tint + red bleed composite into muddy brown. The spec needs a blend mode or an opacity ceiling for when `--song-color` is chromatically distant from the teal base.

**R3. The first-play reveal of the per-song pipeline.** The default `--song-color` is `oklch(0.55 0.15 185)` — the same hue family as the shader's fixed `uColor1` (`oklch(0.72 0.18 185)`). The app looks identical with and without music until the first song plays. On first play, the user sees the feature activate — but they don't know what it looked like before, so the contrast is lost. A slightly warm or cool default (e.g., `oklch(0.55 0.12 210)` — cooler blue) would make the first song transition more apparent.

**R4. JetBrains Mono adoption scope.** The spec says "JetBrains Mono for timestamps, file format badge, query syntax, tag chips, duration." The current codebase's `--font-mono` is applied to: query input (`QueryInput.tsx:245`), operator keys (`mix-kbd` uses `ui-monospace`), timestamps (tabular-nums on formatDuration spans), and file format badges. A font swap at the `--font-mono` token level would be one line, but the `mix-kbd` class hardcodes `ui-monospace` directly — it won't pick up the change unless its CSS is also updated.

Qwen's review (`docs/design/qwen3.7 op.md`, 158 lines) focuses on C-extension stability, GPU thermal behavior, and color gamut safety — angles no other reviewer (including me) covered. One claim is factually wrong (`useEffectEvent` was not removed from React 19.2), but the practical advice is correct. Below are the 6 new findings and why each was missed:
