## Playlist Flash Bug — Root Cause

### Symptom
When clicking on any playlist in the sidebar, all playlist items briefly disappear and are replaced by skeleton placeholders, then reappear after the playlist content loads.

### Root Cause
A **single `loading` boolean** in `playlistStore` is shared between two operations:

1. **`fetchPlaylists()`** — loads the playlist list for the sidebar
2. **`fetchPlaylistDetail(id)`** — loads the songs within a specific playlist

When the user clicks a playlist:
1. `PlaylistDetail.tsx` mounts → `useEffect` calls `fetchPlaylistDetail(id)` 
2. `fetchPlaylistDetail` sets `{ loading: true }` in playlistStore (line 50 of `playlistStore.ts`)
3. **Sidebar.tsx** subscribes to `usePlaylistStore((state) => state.loading)` (line 53) 
4. Sidebar sees `loading: true` → renders skeleton placeholders instead of the playlist list (lines 129-134)
5. API call completes → `loading: false` → playlists reappear

### Fix
Add a separate `detailLoading` state to playlistStore:

```typescript
// playlistStore.ts — add to state interface:
detailLoading: boolean  // isolated from fetchPlaylists' loading

// fetchPlaylistDetail — set detailLoading instead of loading:
set({ detailLoading: true, error: null })
// ... API call ...
set({ activePlaylist: res.data, detailLoading: false })

// fetchPlaylists — unchanged, continues using loading

// Sidebar.tsx — unchanged, reads loading (only set by fetchPlaylists)
// PlaylistDetail.tsx — reads detailLoading instead of loading
```

**Files to change:**
- `frontend/src/stores/playlistStore.ts` — add `detailLoading` state, update `fetchPlaylistDetail`
- `frontend/src/components/playlists/PlaylistDetail.tsx` — read `detailLoading` (line 53 → `usePlaylistStore(state => state.detailLoading)`)

**Estimated effort:** Small (S) — ~10 lines changed across 2 files

## Phase 2: Core Music Player Features (v1.0 Essentials)

### Task 2.1: True Gapless Playback (Pre-buffer)
- **Files:** `frontend/src/hooks/useAudioPlayer.ts`
- **Description:** Building on Task 1.3 (B4 fix), implement proper gapless:
  - Pre-create the next Howl and start loading its audio buffer while current song plays
  - When current song ends (or crossfade triggers), swap to the pre-loaded Howl without silence gap
  - Howler.js limitation: can't fully pre-buffer programmatically. Strategy:
    - Create next Howl with `preload: true` during the last 5s of current song
    - Use the `onload` callback to confirm buffer ready
    - On transition, call `nextHowl.play()` and `currentHowl.stop()` in same frame
    - Use HTML5 audio's `AudioContext` to schedule precise start times if possible
  - Fallback: if next song isn't preloaded in time, current crossfade behavior kicks in
- **Agent:** frontend (audio specialist)
- **Parallel:** No dependencies (independent of other Phase 2 tasks)
- **Effort:** L

### Task 2.2: ReplayGain Support
- **Files:** `backend/app/services/file_scanner.py`, `backend/app/models.py`, `backend/app/database.py`, `backend/app/routers/songs.py`, `frontend/src/types/index.ts`, `frontend/src/hooks/useAudioPlayer.ts`
- **Description:** 
  - **Backend:** During scan, extract ReplayGain tags from audio files using mutagen:
    - MP3: `TXXX:replaygain_track_gain`, `TXXX:replaygain_track_peak`
    - FLAC/Vorbis: `REPLAYGAIN_TRACK_GAIN`, `REPLAYGAIN_TRACK_PEAK`, `REPLAYGAIN_ALBUM_GAIN`, `REPLAYGAIN_ALBUM_PEAK`
    - M4A/MP4: `----:com.apple.iTunes:replaygain_track_gain`
    - Store as `replaygain_track_gain` (float dB), `replaygain_track_peak` (float), `replaygain_album_gain`, `replaygain_album_peak` in songs table
  - **DB Migration:** Add 4 new nullable REAL columns to songs table
  - **API:** Include ReplayGain fields in SongResponse model
  - **Frontend:** 
    - `useAudioPlayer` applies gain using Howler's `volume()` as a multiplier on the stored volume
    - Formula: `adjustedVolume = volume * 10^(replaygain/20)`, clamped to Howler's volume range (0-1)
    - Caveat: Howler.js doesn't have a native gain node we can insert before the audio graph (uses `html5:true` which bypasses Web Audio API for the source)
    - Alternative: Use Web Audio API `GainNode` directly by creating a `MediaElementAudioSourceNode` from Howler's internal audio element, then connect through a gain node. This requires accessing Howler's internal `_sounds[0]._node` (fragile).
    - **More robust alternative:** Use `useAudioAnalyser`'s existing `MediaElementAudioSourceNode` and insert a `GainNode` between source and destination. Set gain based on ReplayGain + user volume.
  - Add ReplayGain mode selector in Settings: Off / Track / Album (default: Track)
- **Agent:** fullstack (backend migration + frontend audio integration)
- **Parallel:** Independent of other Phase 2 tasks
- **Effort:** L
